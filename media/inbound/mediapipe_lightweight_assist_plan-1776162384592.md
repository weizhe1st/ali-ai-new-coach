# MediaPipe 轻量辅助化改造方案（MD版）
## 适用阶段
当前阶段：阿里云版 AI 网球教练系统已完成主链路打通，知识库检索、黄金标准对比、统一回复层已基本完善。  
目标：将 MediaPipe 从“主分析引擎”降级为“关键帧量化辅助模块”。

---

## 一、核心结论

当前不建议继续把 MediaPipe 作为**主分析引擎**来优化。  
更适合的定位应当是：

```text
Qwen-VL / 视觉大模型：负责主判断
MediaPipe：负责关键帧量化补充
知识库 + 黄金标准：负责规则解释与训练建议
```

也就是说：

- **主判断**：由视觉大模型完成
- **量化补充**：由 MediaPipe 对少量关键帧完成
- **最终报告**：由视觉分析 + 知识库 + 黄金标准 + 少量量化指标联合生成

---

## 二、为什么不建议继续把 MediaPipe 作为主引擎

### 1. 当前服务器条件不适合全量跑姿态检测
你当前环境中：
- 服务器无 GPU
- CPU 模式下 MediaPipe 检测效率低
- 全帧/高频采样成本高
- 实际收益有限

例如你已经观察到：
- 50 帧里只检测到 4 帧有效结果

这说明继续把 MediaPipe 作为主引擎，会长期陷入：
- 采样率问题
- 性能问题
- 漏检问题
- 兼容问题

---

### 2. 你当前系统主链路已经被 Qwen-VL 驱动起来了
目前已经具备：
- 真实视频主链路
- 分析服务
- 知识库检索
- 黄金标准对比
- 统一报告输出

所以当前主价值已经不再是“多跑一点骨骼点”，  
而是：
- 报告更准
- 解释更像教练
- 对问题定位更稳定

---

### 3. MediaPipe 更适合输出“量化证据”
它最适合做这些事情：
- 关键帧角度量化
- 抛球手高度估计
- 击球点相对高度估计
- 身体倾斜、肩髋旋转趋势估计

这些是“补充证据”，而不是“主裁判”。

---

## 三、推荐的最终架构

建议把视频分析流程整理成：

```text
真实视频
  -> Qwen-VL 识别关键阶段与关键时刻
    -> 选出少量关键帧
      -> MediaPipe 只跑这些关键帧
        -> 输出量化指标
          -> 知识库检索 + 黄金标准对比
            -> 最终报告生成
```

### 模块分工建议

#### 1. Qwen-VL
负责：
- 判断动作类型（当前优先发球）
- 识别动作阶段
- 识别关键问题
- 选出关键时刻
- 给出主结论

#### 2. MediaPipe
负责：
- 对关键时刻的少量帧做姿态估计
- 输出量化辅助信息
- 给主结论提供数值补充

#### 3. 知识库检索
负责：
- 按阶段检索
- 按问题标签匹配
- 输出针对性教练知识

#### 4. 黄金标准对比
负责：
- 将当前动作与不同 NTRP 水平标准比较
- 生成优先改进项

#### 5. 报告生成器
负责：
- 整合视觉结论 + 量化补充 + 知识库 + 黄金标准
- 输出清晰报告

---

## 四、推荐优化路线

## 路线 A：Qwen-VL 选关键时刻，MediaPipe 只打关键帧（最推荐）

### 目标
让视觉大模型先选出真正值得跑姿态检测的少量帧，  
再让 MediaPipe 只处理这些帧。

### 适合原因
- 最省 CPU
- 最符合当前架构
- 最容易提升稳定性
- 不需要大改主链路

### 建议做法
Qwen-VL 先输出：
- ready 帧
- toss peak 帧
- trophy / loading 帧
- contact 帧
- follow-through 帧

然后每个阶段只取：
- 1~3 帧

总量大概控制在：
- **5~12 帧**

这样 MediaPipe 的工作量会从：
- 全视频采样 30~50 帧  
降到：
- 仅 5~12 帧

---

## 路线 B：降低采样频率（可作为备选）

如果暂时还没把“关键阶段选择”做得很好，也可以先做一个中间版本：

### 做法
- 每 5 帧取 1 帧
- 或每 10 帧取 1 帧
- 只在视频前半段 / 中后段挑关键区间

### 优点
- 实现简单
- 可以快速降低 CPU 压力

### 缺点
- 仍然会跑很多无效帧
- 不如关键帧方案聪明

---

## 路线 C：使用 Lite 模型（可选）

如果 MediaPipe 当前模型支持轻量版，可以尝试：
- PoseLandmarker Lite
- 更低分辨率输入
- 更严格的检测时机

### 优点
- 更快
- 适合 CPU

### 缺点
- 精度可能进一步下降
- 如果关键帧本来就少，收益未必特别大

---

## 五、当前最不推荐的方向

### 1. 不建议继续做“全视频全量 MediaPipe 优化”
例如：
- 一直想办法提升 50 帧里有效检测数
- 针对 CPU 模式不断调参
- 追求 MediaPipe 输出更多帧

原因：
- 算力成本高
- 稳定性不一定提升
- 不符合你当前系统已经形成的“视觉大模型主导”方向

---

### 2. 不建议为了 MediaPipe 现在就换 GPU 服务器
当前更优先的不是提升骨架检测规模，而是：
- 保证主链路稳定
- 提升报告质量
- 建立黄金样本沉淀能力

等以后需要做：
- 更高精度量化评分
- 更复杂动作拆解
- 大规模自动评估

再考虑 GPU 更合适。

---

## 六、建议 MediaPipe 现在负责的量化指标

当前阶段建议只做少量、高价值指标，不要一上来就求全。

### 1. 抛球相关
- 抛球手臂伸展程度
- 抛球高度相对头顶位置
- 抛球偏前 / 偏后趋势

### 2. 加载阶段相关
- 肩髋角度趋势
- 躯干倾斜趋势
- 下肢加载是否明显

### 3. 击球阶段相关
- 击球点相对高度
- 击球时身体是否完全顶起
- 手臂伸展趋势

### 4. 随挥阶段相关
- 随挥是否完整
- 身体是否前送
- 落地稳定性

---

## 七、建议输出形式

MediaPipe 不要直接输出“大段结论”，而应输出结构化量化结果，例如：

```json
{
  "mediapipe_metrics": {
    "toss_arm_extension_score": 0.72,
    "contact_height_score": 0.64,
    "shoulder_rotation_score": 0.58,
    "follow_through_completeness_score": 0.81
  },
  "frame_measurements": [
    {
      "stage": "toss",
      "frame_index": 102,
      "arm_extension_angle": 154.2
    },
    {
      "stage": "contact",
      "frame_index": 188,
      "contact_relative_height": 0.83
    }
  ]
}
```

这样：
- 报告生成器可直接引用
- 知识库检索可做标签辅助
- 黄金标准对比可更容易使用

---

## 八、建议与报告生成的结合方式

报告中不要把 MediaPipe 放在第一优先级，而应作为“量化补充”。

### 推荐写法
例如：

```text
视觉判断：
抛球不够稳定，击球点略偏后。

量化辅助：
在关键帧检测中，抛球手臂伸展角度偏小，击球相对高度低于理想区间。

综合建议：
优先修正抛球路径和击球点管理，再提升整体发力效率。
```

这样更自然，也更稳。

---

## 九、建议的实施顺序

### 第一步：不改主链路
先保持当前：
- 视频输入
- AnalysisService
- 知识库检索
- 黄金标准对比
- 报告生成

都不动。

### 第二步：加关键时刻输出
让 Qwen-VL 或当前分析服务先输出：
- ready
- toss
- loading
- contact
- follow  
对应的关键帧索引或时间点。

### 第三步：MediaPipe 只吃关键帧
新增一个轻量模块：
- 输入：少量关键帧
- 输出：量化指标

### 第四步：报告生成器引用量化结果
在最终报告中加入：
- 少量量化补充
- 不取代主视觉结论

---

## 十、建议当前阶段一句话结论

**当前最优路线不是继续把 MediaPipe 做成主分析引擎，而是让 Qwen-VL 主导阶段识别与问题判断，再让 MediaPipe 只对关键帧做量化补充。**

---

## 十一、最终建议

如果你现在只选一个方向推进，我建议你选：

### **Qwen-VL 主导 + MediaPipe 关键帧辅助**

这是最符合你当前系统状态的方案，因为它：
- 不破坏现有主链路
- 最省 CPU
- 最容易落地
- 最符合“教练式报告 + 少量量化证据”的目标

---

## 十二、一句话行动建议

**下一步不要优先优化 MediaPipe 的全量检测率，而应先让系统输出关键阶段帧，再让 MediaPipe 只跑这些关键帧。**
