# 真实钉钉 / QQ 小范围试运行检查清单（MD版）
## 适用阶段
当前版本：`v0.1-internal-trial`  
目标：在**不继续大改架构**的前提下，对真实渠道做小范围试运行验证。

---

## 一、试运行目标

本轮试运行不追求“大规模上线”，只验证三件事：

1. **真实消息能否进入主链路**
2. **真实视频能否被正确落盘并分析**
3. **失败时能否给出明确、可排障的反馈**

---

## 二、试运行顺序

建议按以下顺序进行：

### 第一阶段：钉钉文字消息
验证：
- 真实钉钉消息是否能进入 adapter
- 是否能生成 `UnifiedMessage`
- 是否能进入 `MessageRouter`
- 是否能生成统一 reply
- 是否能正常回给钉钉

### 第二阶段：钉钉视频消息
验证：
- 视频是否真实进入任务
- 是否生成 `task_id`
- 是否写入 `source_file_path`
- 是否进入 `AnalysisService`
- 是否收到统一报告回复

### 第三阶段：QQ 文字消息
验证：
- 是否复用同一主链路
- 是否不是单独一套旧逻辑
- 回复格式是否与钉钉一致

### 第四阶段：QQ 视频消息
验证：
- 视频是否能走同一输入准备层
- 是否能生成统一任务与统一回复
- 是否没有偷偷走旧入口脚本

---

确认以下目录可写：
- `data/`

确认服务器运行版本与 GitHub 试运行版本一致。

建议执行：

```bash
git rev-parse HEAD
git tag
```

## 四、钉钉试运行检查项

### A. 文字消息
发送一条普通文字，例如：
- “你好”
- “测试一下”
- “请分析”

应检查：
- 钉钉收到回复
- 日志中能看到消息进入 adapter
- 能生成 `task_id`
- reply 来自 `ReplyBuilder`
- 无异常报错

### B. 视频消息
发送一个小视频样本，建议：
- 时长短
- 文件路径正常
- 文件名清晰

应检查：
- 生成独立任务目录
- `source_file_name` 正确
- `source_file_path` 正确
- `preparing_video_input` 阶段正常
- `executing_video` 阶段正常
- 最终收到分析回复

### C. 错误验证
故意发送一个异常视频场景，例如：
- 空文件
- 无法解析的视频
- 路径缺失场景

应检查：
- 任务失败
- 错误码明确
- 不会 fallback 到默认样例视频
- 钉钉收到统一失败回复

---

## 五、QQ 试运行检查项

### A. 文字消息
发送普通文字，确认：
- 使用同一 `UnifiedMessage`
- 使用同一 `MessageRouter`
- 使用同一 `ReplyBuilder`
- 回复文案逻辑与钉钉一致

### B. 视频消息
发送一个真实视频，确认：
- 进入同一视频输入准备层
- 使用同一 `AnalysisService`
- 使用同一统一回复层
- 没有额外单独 QQ 视频分析逻辑

### C. 错误验证
验证异常场景时，确认：
- QQ 也能收到统一失败回复
- 错误结构与钉钉一致
- 只是渠道展示方式不同，不是业务逻辑不同

---

## 六、必须重点盯住的日志

试运行时，至少确认日志里能看到这些关键信息：

### 1. 消息进入
- 渠道名
- 用户标识
- 消息类型
- task_id

### 2. 视频输入准备
- source_file_url
- source_file_name
- source_file_path
- 工作目录

### 3. 分析执行
- 是否进入 `AnalysisService`
- 是否成功返回
- 是否失败
- 错误码是什么

### 4. 回复输出
- 是否进入 `ReplyBuilder`
- 最终 reply 是否成功生成
- 回复是否成功发出

---

## 七、试运行样本建议

建议最少准备以下 4 类样本：

### 样本 1：正常文字
用途：验证文字主链路

### 样本 2：正常短视频
用途：验证标准视频主链路

### 样本 3：异常视频
用途：验证失败回复和错误码

### 样本 4：边界视频
例如：
- 文件名特殊
- 时长偏短
- 路径来源异常

用途：验证输入准备层是否稳

---

## 八、试运行通过标准

满足以下条件，可认为进入“小范围内部试运行可用”：

### 必须满足
- 钉钉文字链路通过
- 钉钉视频链路通过
- QQ 文字链路通过
- QQ 视频链路至少基本通过
- 视频输入失败场景处理正确
- 不存在默认样例 fallback
- 日志能追踪 task_id 和 source_file_path
- 回复格式统一
- 生产环境变量已配置完整

### 暂不要求
- 大规模并发
- 自动重试
- 富文本卡片
- 全自动部署
- 完整监控平台

---

## 九、发现问题时的处理顺序

如果试运行中出现问题，请按以下顺序排查：

### 1. 先看消息是否进入 adapter
如果没进入，先排查渠道配置

### 2. 再看是否生成 `UnifiedMessage`
如果没有，排查 payload 解析

### 3. 再看是否生成 `UnifiedTask`
如果没有，排查 router

### 4. 再看视频是否进入输入准备层
如果没有，排查 executor

### 5. 再看 `source_file_path` 是否真实存在
如果不存在，排查视频输入准备逻辑

### 6. 再看 `AnalysisService`
如果失败，排查模型配置、analysis backend、API Key

### 7. 再看 `ReplyBuilder`
如果分析成功但回复异常，排查统一回复层

## 十一、最重要的判断标准

这一轮不要只看“有没有回复”，要重点看：

**回复的这份结果，到底是不是针对用户刚刚发来的那个真实视频产生的。**

只要这一点能稳定确认，你这套系统就已经进入一个非常关键的可用阶段了。
