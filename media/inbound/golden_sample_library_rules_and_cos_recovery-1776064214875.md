# 黄金样本库与 COS 接入恢复建议（MD版）
## 适用阶段
当前阶段：阿里云版 AI 网球教练系统已完成主链路打通，进入小范围内部试运行阶段。  
本文目标：明确黄金样本库入库规则、COS 是否应重新接入、以及迁移过程中建议补回的能力。

---

## 一、当前最关键的判断

### 1. 现在能不能重新打开 COS 上传功能？
**可以考虑重新打开，但不建议直接恢复成“只传 COS、不落本地”的旧模式。**

当前更稳妥的方式应当是：

```text
真实上传视频
  -> 先进入本地任务链路
    -> 本地完成入链、分析、日志、回复
      -> 再异步/同步复制到 COS
```

也就是说：

- **本地链路继续作为当前主执行链路**
- **COS 作为存储归档层 / 样本沉淀层 / 后续扩展层**
- 不建议现在就把 COS 重新变成“唯一输入源”或“唯一分析源”

---

### 2. 为什么现在可以考虑恢复 COS？
因为当前主链路已经验证通过：
- 钉钉文字：通过
- 钉钉视频：通过
- QQ 文字：通过
- QQ 视频：通过
- 视频真实入链：通过
- `source_file_path` 对应真实视频：通过
- 分析与统一回复链路：通过

这说明现在你已经有条件把 COS 作为“第二层存储”加回来，而不是像之前那样在主链路不稳时就强行依赖 COS。

---

### 3. 最稳妥的恢复策略是什么？
建议采用：

```text
本地优先 + COS 归档
```

而不是：

```text
COS 优先 + 本地兜底
```

原因：
- 现在本地主链路已经跑通
- 本地链路更容易排障
- COS 更适合作为沉淀、归档、样本复用
- 这样能把你以前已上传的黄金样本与新样本慢慢合并起来

---

## 二、黄金样本库入库规则（重新整理版）

### 1. 规则目标
黄金样本库不是普通视频仓库，也不是所有能分析成功的视频都自动进入。  
它的作用是：

1. 作为高质量动作参考样本
2. 作为后续知识库提炼依据
3. 作为规则校正、分析对照、模型优化的基准数据
4. 避免低质量、错误动作、遮挡严重、分析不稳定的视频污染标准

---

### 2. 当前建议原则

#### 原则 1：先不要全自动入库
当前阶段**不建议**把“分析成功”的视频直接自动放进黄金样本库。

原因：
- 分析成功 ≠ 样本质量合格
- 视频清晰 ≠ 动作标准
- 动作标准 ≠ 适合作为黄金样本
- 一旦黄金样本库混入低质量视频，后续会污染规则和知识库

#### 原则 2：先采用“两层机制”
建议当前采用：

```text
真实上传视频
  -> 分析完成
    -> 候选样本池
      -> 人工复核
        -> 黄金样本库
```

也就是说：

- **第一层：候选样本池**
- **第二层：黄金样本库**

---

### 3. 候选样本池准入条件

只有先满足基础条件，视频才可以进入“候选样本池”。

#### 技术条件
必须满足：
- 视频文件完整可播放
- 能成功进入分析链路
- 能成功生成 `task_id`
- 能正确写入 `source_file_path`
- 分析过程无异常中断
- 报告生成完整

#### 内容条件
必须满足：
- 主体人物清晰
- 动作主体基本完整出现在画面中
- 关键动作阶段可见
- 遮挡不严重
- 视频不是明显无关内容

#### 来源条件
建议满足：
- 来源明确
- 任务日志可追踪
- 能确认不是默认样例或错误串任务产物
- 能确认是用户真实上传的视频

#### 动作类型条件
当前阶段建议只纳入：
- **发球动作**

其他动作（正手、反手、截击等）建议暂不进入黄金样本库，除非后续单独扩展。

---

### 4. 不能进入候选样本池的情况

出现以下任一情况，不进入候选池：

- 视频损坏或无法播放
- 视频与任务不匹配
- 任务串号
- 分析失败
- 使用了默认样例视频
- 画面遮挡严重
- 主体动作无法辨认
- 视频不是发球动作
- 视频过短或关键动作缺失
- 多人混杂，无法确认分析对象
- 报告明显异常或严重失真

---

### 5. 黄金样本库正式入库条件

候选样本池中的视频，不应自动进入黄金样本库。  
正式入库建议满足以下条件：

#### 画面质量合格
- 清晰度基本可用
- 光线不过暗
- 主体不严重模糊
- 关键动作可辨认

#### 动作可学习性强
- 发球动作阶段清楚
- 抛球、引拍、加载、击球、随挥可观察
- 有足够参考价值
- 不属于明显错误到无法作为参考的动作

#### 分析结果稳定
- 多次观察结果大体一致
- 报告逻辑正常
- 关键问题提取合理
- 没有明显误识别

#### 样本用途明确
建议至少属于以下之一：
- 优秀示范样本
- 典型问题样本
- 可用于规则校正的边界样本
- 可用于训练建议模板提炼的代表样本

#### 人工确认通过
必须由你或教练 / 标注人员明确确认：
- “同意入黄金样本库”

---

### 6. 黄金样本建议分类

黄金样本库建议至少分为：

#### A. 优秀动作样本
用于：
- 正向参考
- 标准动作对照
- 训练目标示例

#### B. 典型问题样本
用于：
- 展示常见错误
- 规则解释样本
- 提示语或建议模板依据

#### C. 边界样本
用于：
- 测试系统鲁棒性
- 检查模型误判
- 检查规则是否过严或过宽

---

### 7. 建议记录的入库元数据

每个进入黄金样本库的视频，建议至少记录以下元数据：

```json
{
  "sample_id": "golden_0001",
  "task_id": "xxx",
  "action_type": "serve",
  "sample_category": "excellent_demo",
  "source_file_name": "xxx.mp4",
  "source_file_path": "xxx",
  "cos_key": "golden/serve/excellent/xxx.mp4",
  "analysis_summary": "xxx",
  "score": 78,
  "ntrp_level": "3.5",
  "tags": ["toss", "timing", "loading"],
  "review_status": "approved",
  "reviewer": "xxx",
  "reviewed_at": "2026-04-13T13:00:00Z"
}
```

至少推荐保留：
- `sample_id`
- `task_id`
- `action_type`
- `sample_category`
- `source_file_name`
- `source_file_path`
- `cos_key`
- `analysis_summary`
- `tags`
- `review_status`
- `reviewer`
- `reviewed_at`

---

### 8. 当前阶段最稳妥的入库流程

建议采用下面这个流程：

```text
用户上传视频
  -> 系统分析完成
    -> 满足基础技术条件
      -> 标记 candidate_for_golden = true
        -> 人工复核
          -> approve / reject
            -> 如果 approve，则复制/归档到黄金样本库（本地+COS）
```

建议增加字段：
- `candidate_for_golden`
- `golden_review_status`
- `golden_review_note`

例如：
- `candidate_for_golden = true`
- `golden_review_status = pending`
- `golden_review_note = "动作完整，建议人工复核"`

## 三、现在是否应该恢复 COS：我的建议

## 结论
**建议恢复，但按“分阶段恢复”来做。**

---

### 阶段 1：本地主链路不变，分析完成后再上传 COS
推荐流程：

```text
真实视频
  -> 本地下载 / 入链
  -> 本地分析
  -> 本地生成报告
  -> 如果成功，则上传 COS
  -> 记录 cos_key / cos_url
```

#### 这一阶段的目标
- 不破坏现有稳定主链路
- 让新视频开始沉淀到 COS
- 与以前已在 COS 的黄金样本逐步汇合

#### 这一阶段最适合你
因为当前最重要的是：
- 先保证系统稳定
- 再恢复长期存储与样本沉淀

---

### 阶段 2：候选样本自动归档到 COS 指定目录
例如：

```text
cos://bucket/
  inbound/
  analyzed/
  candidate_golden/
  golden/
```

建议按这四层管理：

#### `inbound/`
原始上传视频暂存

#### `analyzed/`
成功分析完成的视频

#### `candidate_golden/`
满足候选条件、待人工审核的视频

#### `golden/`
人工审核通过后的正式黄金样本

---

### 阶段 3：把历史黄金样本与新样本统一索引
你之前 COS 上已有部分黄金样本视频，所以后续应补一层：

- 历史黄金样本登记表
- 新增黄金样本登记表
- 统一 sample_id
- 统一 category / tags / reviewer / review_status

这样以后才能真正把旧样本和新样本放到一起使用，而不是只是“同一个桶里堆文件”。

---

## 四、为什么现在不建议直接恢复“COS 主输入模式”

不建议现在就改成：
- 用户上传 -> 直接走 COS -> 再分析

原因：

### 1. 你刚刚打通的是本地主链路
现在本地：
- 接收
- 入链
- 分析
- 回复
已经真实验证通过

如果这时候再把输入主链改成 COS 优先，会增加：
- 下载链路复杂度
- 权限问题
- URL 时效问题
- 排障难度

### 2. 你当前目标是试运行，不是彻底重构
当前最稳妥的是：
- 先跑通
- 再沉淀
- 再优化

不是再马上改输入主链。

### 3. COS 更适合当前作为“沉淀层”
你现在最需要的是把新视频和旧黄金样本逐步归拢，而不是让 COS 重新控制整条主链路。

## 五、迁移过程中大概率被省略、建议逐步补回的能力

下面这些功能，很多在旧系统里可能“部分存在”或“隐式存在”，但在本次迁移和打通主链时被弱化或省略了。建议你逐步补回来。

建议补回：
- 分析成功后自动上传 COS
- 保存 `cos_key`
- 保存 `cos_url`
- 本地 / COS 双份可追踪

#### 优先级
**高**

因为你已经明确有历史黄金样本在 COS 上。

---

### 2. 黄金样本候选标记能力
建议补回：
- 分析完成后判断是否进入 candidate pool
- 自动写 `candidate_for_golden=true/false`
- 自动记录候选理由

#### 优先级
**高**

因为这直接关系到样本沉淀。

---

### 3. 人工复核与正式入库能力
建议补回：
- 人工确认通过 / 不通过
- 审核备注
- 审核人
- 审核时间
- 审核通过后进入 `golden/`

#### 优先级
**高**

因为黄金样本库不适合全自动入库。

---

### 4. 样本元数据索引能力
建议补回：
- task_id
- source_file_path
- cos_key
- 分析摘要
- 标签
- reviewer
- review_status

- JSON

都行，不一定一上来就大数据库。

#### 优先级
**高**

### 5. 历史黄金样本导入整理

#### 优先级
**高**

### 6. 失败任务重试 / 重跑能力

**中**

### 7. 任务查询 / 后台检索能力

#### 优先级
**中**

### 8. 报告沉淀与回看能力
建议补回：

#### 优先级
**中**

### 9. 管理后台 / 运维页面
这个当前可以先不急，但未来建议有：
- 任务状态

**中低**

### 10. 多动作扩展能力

**后续**
