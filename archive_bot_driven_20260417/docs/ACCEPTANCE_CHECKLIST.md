## 二、需验证链路

### 1. 文字任务链路

**流程**:
```
钉钉/QQ 文字消息
  → Adapter (parse_xxx_message)
  → UnifiedMessage
  → MessageRouter.route_message()
  → TaskExecutor.execute()
  → ReplyBuilder.build_reply()
  → 渠道输出
```

**验收点**:
- [ ] 能生成 UnifiedMessage
- [ ] 能生成 UnifiedTask
- [ ] 能返回统一 reply
- [ ] 状态标记正确
- [ ] 回复文案来自 ReplyBuilder

### 2. 视频任务链路

**流程**:
```
钉钉/QQ 视频消息
  → Adapter
  → UnifiedMessage
  → MessageRouter
  → TaskExecutor
  → VideoInputHandler (准备视频输入)
  → AnalysisService (调用分析)
  → ReplyBuilder
  → 渠道输出
```

**验收点**:
- [ ] 视频工作目录生成正确
- [ ] source_file_path 被写入
- [ ] AnalysisService 使用标准输入
- [ ] ReplyBuilder 输出统一
- [ ] 不再直接依赖旧 reply 逻辑

### 3. 失败场景处理

**场景 A: 视频输入缺失**
- [ ] 任务失败
- [ ] 错误码清晰
- [ ] 回复层给出统一失败回复
- [ ] 不允许 fallback 到默认样例

**场景 B: 分析服务失败**
- [ ] TaskExecutor 能捕获异常
- [ ] 状态标记为 failed
- [ ] 回复层给出统一错误回复

**场景 C: 配置缺失**
- [ ] 配置层或分析层明确失败
- [ ] 不允许静默退化
- [ ] 错误信息可排查

### 4. 配置层验证

- [ ] config.py 可加载 dev 模式配置
- [ ] config.py 可加载 prod 模式配置
- [ ] analysis_service.py 使用配置层
- [ ] adapter 不再直接散读环境变量
- [ ] 路径配置会正确生成工作目录
- [ ] .env.example 与代码字段一致

### 5. 状态、日志、回复协同

**文字任务**:
- [ ] 状态流转正常
- [ ] 日志可记录执行开始/成功/失败
- [ ] reply builder 输出与状态一致

**视频任务**:
- [ ] preparing_video_input → executing_video → completed/failed
- [ ] 状态、日志、回复三者不矛盾

| **数据库未初始化脚本** | app.db 需手动创建 | 部署复杂 | 添加初始化脚本 |

| 风险项 | 说明 | 影响 | 建议 |
